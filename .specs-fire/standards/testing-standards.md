# Testing Standards

## Overview

No test framework is currently configured in the project. This document captures the recommended approach for when tests are added.

## Testing Framework

**Framework**: Not configured (recommended: Vitest for renderer, Jest or Vitest for main)
**Runner**: Not configured

## Test Types

| Type | Tool | Location | When to Use |
|------|------|----------|-------------|
| Unit | Vitest (recommended) | `src/**/__tests__/` | Pure functions, utilities, store logic |
| Integration | Vitest (recommended) | `src/main/__tests__/` | IPC handlers + real SQLite DB |
| E2E | Playwright + Electron | `e2e/` | Critical user flows |

## Coverage Requirements

**Target**: 60%
**Enforcement**: Not enforced (CI not configured)

**Critical paths that MUST have coverage:**
- IPC handler happy paths and error paths
- DB repository CRUD operations
- LLM prompt builder output
- Agent runner orchestration logic

## Test Naming

**Pattern**: `describe('<module>', () => { it('<behavior>', ...) })`

**Examples**:
- `it('returns error response when DB insert fails')` — error path
- `it('builds prompt with system context and history')` — happy path
- `it('routes to correct skill when intent matches')` — routing logic

## Test Structure

```typescript
describe('ChatRepository', () => {
  let db: Database

  beforeEach(() => {
    db = openInMemoryDb()
    runMigrations(db)
  })

  afterEach(() => {
    db.close()
  })

  it('inserts and retrieves a message', () => {
    const repo = new ChatRepository(db)
    repo.insertMessage({ roomId: '1', role: 'user', content: 'hello' })
    const messages = repo.getMessages('1')
    expect(messages).toHaveLength(1)
    expect(messages[0].content).toBe('hello')
  })
})
```

## Mock Strategy

**Approach**: Integration-first for main process (use real in-memory SQLite). Mock only external boundaries (LLM HTTP calls, OS APIs).

**Guidelines**:
- Use real SQLite in-memory DB for repository tests — don't mock the DB
- Mock `openai` HTTP calls with `vi.mock` or MSW
- Don't mock internal modules unless they cross a process boundary
- IPC handlers should be tested with real handler functions, not mocked

## Test Data

**Strategy**: Inline factory functions or fixture objects per test file.

**Guidelines**:
- Keep test data close to the test that uses it
- Use realistic but minimal data — don't over-specify unrelated fields
- Avoid shared mutable fixtures across tests

## Running Tests

```bash
# Run all tests
npx vitest run

# Run with coverage
npx vitest run --coverage

# Run specific test file
npx vitest run src/main/__tests__/chat-repository.test.ts

# Run in watch mode
npx vitest
```
